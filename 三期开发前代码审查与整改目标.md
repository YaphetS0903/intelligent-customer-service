# 三期开发前代码审查与整改目标

## 1. 文档目的

本轮审查用于确认当前系统在进入三期系统集成开发前的代码质量、生产安全和数据一致性情况。

本文件最初记录三期开发前发现的鉴权边界、审计数据删除、并发一致性、依赖漏洞和重复功能等问题。2026-07-14 已完成本轮 P0、P1 整改和关键功能收敛，并在独立测试数据库完成完整回归；详细结果见第 12 节。

## 2. 审查范围

- 用户登录、注册、SSO、LDAP 和会话认证
- 管理端页面及 API 权限边界
- 知识库上传、OCR、Excel 和 PDF 解析
- 智能问答、会话、反馈和人工工单
- 培训课程、考试、证书和学习记录
- TTS、数字人和第三方服务调用
- MySQL、Supabase 和内存数据库适配
- 运营看板、审计、部署检查和运维页面
- 自动化测试、生产构建和依赖安全

## 3. 总体结论

### 当前状态

- `npm run typecheck`：通过
- `npx tsc --noEmit --noUnusedLocals --noUnusedParameters`：通过
- `npm run build`：通过
- Playwright 独立测试数据库回归：`26/26` 通过
- 生产依赖审计：高危 `0`、严重 `0`，剩余 Next.js/PostCSS 中危 `2`
- 当前代码已达到三期接口开发准入条件

### 主要风险

1. 生产环境账号注册和管理员降级逻辑存在权限风险。
2. 会话删除会销毁工单、安全事件和模型用量等审计记录。
3. TTS、反馈和工单接口存在资源消耗或数据归属校验缺口。
4. 培训考试缺少事务和幂等控制，并发提交可能产生重复成绩。
5. Excel 解析依赖存在已知高危安全漏洞。
6. 多套数据库实现和多个重复管理页面增加三期开发成本。

## 4. P0：进入三期前必须修复

### 4.1 关闭生产环境自由注册

**现状**

- 登录页面公开展示员工注册入口。
- 任意访问者可以创建状态为 `active` 的员工账号。
- 没有邀请码、管理员审批、企业邮箱限制、验证码或注册限流。

**涉及位置**

- `components/auth-panel.tsx`
- `app/api/auth/register/route.ts`

**整改要求**

- 增加 `ALLOW_SELF_REGISTRATION` 配置，生产环境默认关闭。
- 关闭时隐藏注册入口，并让注册 API 返回明确的禁止注册响应。
- 后续账号来源优先采用管理员创建、LDAP、SSO 或企业微信同步。
- 如保留自助注册，必须增加企业邮箱白名单、审批和限流。

**验收标准**

- 未开启配置时，登录页不显示注册入口。
- 直接调用注册 API 无法创建账号。
- 管理员创建账号和已有账号登录不受影响。

### 4.2 移除默认管理员认证降级

**现状**

- `admin-tianrui` 的数据库查询超时或失败时，会返回硬编码管理员资料。
- 数据库中账号即使已被禁用或删除，旧的有效 Cookie 仍可能获得管理员权限。

**涉及位置**

- `lib/db.ts`

**整改要求**

- 删除 `getDefaultMySqlAdminFallback` 和管理员查询失败降级逻辑。
- 数据库不可用时认证必须失败关闭，不能恢复为默认管理员。
- 管理员状态和角色必须以数据库实时记录为准。

**验收标准**

- 数据库不可用时，后台接口返回服务不可用或认证失败。
- 被禁用管理员的旧 Cookie 无法继续访问管理端。
- 普通员工 Cookie 无法访问任何管理员 API。

### 4.3 替换存在漏洞的 Excel 解析依赖

**现状**

- 当前使用 `xlsx@0.18.5` 在服务端解析员工上传的 Excel。
- 依赖审计命中原型污染和 ReDoS 高危公告。

**涉及位置**

- `package.json`
- `lib/document-text.ts`

**整改要求**

- 优先更换为持续维护且无已知高危漏洞的 Excel 解析方案。
- 增加工作表数量、行数、列数和解析耗时限制。
- 在替换完成前，可临时关闭 `.xls` 和 `.xlsx` 上传。

**验收标准**

- `npm audit --omit=dev` 不再报告当前 Excel 解析高危漏洞。
- 正常 Excel 可以提取文本并进入知识库。
- 超大、损坏或伪装 Excel 能被安全拒绝，不拖垮服务。

### 4.4 重构会话删除与审计保留策略

**现状**

- 删除归档会话会同步删除反馈、工单、工单评论、知识整改任务、安全事件、模型用量和消息。
- MySQL 删除过程由多个独立 SQL 组成，没有事务。
- 删除中途失败可能造成数据只删除一部分。

**涉及位置**

- `lib/mysql-db.ts`
- `lib/db.ts`

**整改要求**

- 员工删除改为软删除或隐藏，不直接物理删除审计数据。
- 安全事件、工单、模型用量和治理记录应按审计保留策略保存。
- 必须物理删除时，由管理员执行，并使用数据库事务。
- 定义会话正文和审计记录各自的保留周期。

**验收标准**

- 员工删除会话后，员工端不再显示该会话。
- 管理员审计中仍可查看必要的工单和安全记录。
- 删除流程任一环节失败时不会出现部分删除。

### 4.5 为 TTS 增加登录校验和资源保护

**现状**

- `/api/tts` 不要求登录。
- 外部访问者可持续消耗自建 TTS 服务器或第三方额度。

**涉及位置**

- `app/api/tts/route.ts`

**整改要求**

- 要求系统登录用户才能调用。
- 增加单次文本长度、并发数量、请求频率和超时控制。
- TTS 服务仍可被其他项目使用，但应通过独立 API Key 或内部网关调用，不能复用本系统公开接口。

**验收标准**

- 未登录调用返回 `401`。
- 登录用户正常播放语音。
- 高频调用不会无限占用 TTS 进程。

### 4.6 增加登录和注册防护

**现状**

- 登录和注册接口没有应用层限流或失败锁定。
- 当前系统通过公网 IP 直接提供访问。

**整改要求**

- 按 IP 和账号增加滑动窗口限流。
- 连续登录失败触发短时间锁定和安全事件。
- 返回统一错误信息，避免泄露账号是否存在。
- 对异常登录写入安全审计记录。

**验收标准**

- 连续错误登录达到阈值后返回 `429` 或临时锁定提示。
- 正常用户在合理频率内不受影响。
- 管理端可查看异常登录事件。

## 5. P1：三期开发前建议完成

### 5.1 修复 HTTP 环境 Cookie 配置不一致

**问题**

- 登录根据 `APP_BASE_URL` 判断是否使用 Secure Cookie。
- 注册、退出和 SSO 根据 `NODE_ENV=production` 强制使用 Secure Cookie。
- 当前 HTTP IP 部署会导致注册或 SSO Cookie 被浏览器拒绝。

**整改要求**

- 提供统一的 Session Cookie 配置函数。
- 所有登录、注册、退出和 SSO 路由使用同一判断规则。
- 上线域名和 HTTPS 后再统一开启 Secure。

### 5.2 修复反馈消息归属校验

**问题**

- 反馈接口接受任意 `message_id`，没有确认消息是否属于当前用户。

**涉及位置**

- `app/api/feedback/route.ts`
- `lib/db.ts`
- `lib/mysql-db.ts`

**整改要求**

- 提交反馈前验证消息、会话和当前用户的归属关系。
- 限制同一用户对同一消息重复创建多条冲突反馈。

### 5.3 修复工单会话和消息归属校验

**问题**

- 员工可以提交任意 `conversation_id` 和 `message_id`。
- 会污染后台工单、通知和会话审计关联。

**涉及位置**

- `app/api/tickets/route.ts`

**整改要求**

- 创建工单前验证会话属于当前用户。
- 如填写消息 ID，必须同时属于该会话。
- 数据库层增加必要的外键或一致性约束。

### 5.4 修复培训考试并发和幂等问题

**问题**

- 开始考试、检查次数、创建成绩和关闭会话不是同一事务。
- 并发开始可能创建多个进行中场次。
- 并发提交可能产生多条成绩并绕过最大考试次数。

**涉及位置**

- `app/api/training/[id]/quiz/route.ts`
- `lib/mysql-db.ts`
- `lib/mysql.ts`

**整改要求**

- 为 `session_id` 的考试成绩增加唯一约束。
- 使用事务和行锁完成会话提交、次数检查和成绩创建。
- 提交接口应具备幂等性，重复提交返回原结果。
- 防止同一用户同一课程同时存在多个有效场次。

**验收标准**

- 并发提交同一场考试只生成一条成绩。
- 最大考试次数在并发场景下仍然有效。
- 每个用户每门课程最多只有一个有效考试场次。

### 5.5 限制运营看板数据访问

**问题**

- `/api/dashboard` 只检查是否登录，没有管理员校验。
- 员工可能取得全局知识库、资料、培训和反馈统计。

**涉及位置**

- `app/api/dashboard/route.ts`
- `lib/dashboard.ts`

**整改要求**

- 管理看板接口仅允许管理员访问。
- 如员工端需要概览，应提供独立且只返回本人数据的接口。
- 修复当前消息数始终为 `0` 的统计错误。

### 5.6 统一管理页面权限保护

**问题**

- `/admin/deploy` 和 `/admin/operations` 页面没有显式调用 `requireAdmin()`。
- 虽然相关 API 多数有管理员校验，但页面权限方式不一致。

**涉及位置**

- `app/admin/deploy/page.tsx`
- `app/admin/operations/page.tsx`

**整改要求**

- 所有 `/admin` 页面统一通过共享管理端布局或守卫校验管理员权限。
- 不再依赖前端菜单隐藏作为权限控制。

### 5.7 修复登录后跳转参数

**问题**

- 普通账号登录完成后直接使用 URL 中的 `next` 参数跳转。

**涉及位置**

- `components/auth-panel.tsx`

**整改要求**

- 只允许以单个 `/` 开头的站内路径。
- 拒绝协议地址、双斜杠和异常编码路径。
- 普通员工不能通过 `next` 进入管理页面。

### 5.8 增强上传文件校验

**问题**

- 当前只要 MIME 或扩展名任一匹配就接受文件。
- 缺少文件签名、压缩包结构和解析资源限制。

**涉及位置**

- `app/api/documents/upload/route.ts`
- `lib/document-text.ts`

**整改要求**

- 校验 PDF、DOCX、PPTX、XLSX 和图片文件签名。
- 限制批量文件数量、解压后大小、页数和解析时间。
- 解析失败时保留清晰状态，不留下不可见的半成品资料。

## 6. P2：重复功能和结构整理

### 6.1 合并重复的知识管理页面

**现状**

- `/admin` 和 `/admin/documents` 都渲染同一个 `AdminDashboard`。
- 菜单分别显示“知识管理”和“资料治理”，实际内容相同。

**建议**

- 保留 `/admin/documents` 作为知识管理主入口。
- `/admin` 重定向到 `/admin/documents`，或改为真正的后台总览。
- 移除重复导航项，减少用户判断成本。

### 6.2 整合运营类页面

**现状**

- 运营工作台、运营看板、运营与审计存在大量指标重叠。

**建议结构**

- 运营总览：核心业务指标、趋势、告警。
- 会话与反馈：会话审计、满意度、热门问题。
- 工单与整改：人工工单、知识缺口、QA 整改。
- 安全审计：异常访问、提示词注入和敏感内容。
- 运维与备份：运行监控、备份、恢复验证。

### 6.3 删除登录页硬编码业务数据

**现状**

- 登录页固定显示 `2` 个知识库、`8` 份资料和 `100%` QA。

**建议**

- 删除这些容易失真的业务数字。
- 登录页只显示产品名称、服务状态和登录入口。

### 6.4 收敛数据库实现

**现状**

- 同时维护 MySQL、Supabase 和内存数据库实现。
- `lib/db.ts` 中存在大量数据库分支。
- Supabase 表结构已经落后于 MySQL，部分安全、监控和 QA 表缺失。

**建议**

- 明确 MySQL 为生产唯一主数据库。
- 将 MySQL 实现拆成领域仓储：用户、文档、会话、工单、培训、审计。
- 内存模式仅保留自动测试或演示用途。
- Supabase 适配标记为废弃或单独维护，不继续混入生产主链路。

### 6.5 拆分超大组件和模块

当前重点文件规模：

- `components/qa-test-admin.tsx`：约 6164 行
- `components/admin-dashboard.tsx`：约 5442 行
- `lib/db.ts`：约 3676 行
- `lib/mysql-db.ts`：约 3168 行
- `components/insights-admin.tsx`：约 2569 行
- `components/chat-client.tsx`：约 2230 行
- `components/settings-wizard.tsx`：约 1973 行

**建议**

- 不进行全项目一次性重写。
- 优先拆分三期会修改的认证、通知、工单、配置和数据库模块。
- 页面组件按标签页、表单和领域功能拆分。
- 数据库函数按领域拆分，并保持对外接口稳定。

### 6.6 清理死代码

严格未使用检查发现的候选项包括：

- 未使用的类型导入
- `components/settings-wizard.tsx` 中未使用的 `envPath`
- `lib/health.ts` 中未使用的 `envReady`
- `lib/local-rag.ts` 和 `lib/provider-http.ts` 中未使用的局部变量

清理时不得顺带重构无关业务逻辑。

## 7. 暂不删除的预留功能

以下能力虽然尚未完整落地，但属于明确的发展方向，不应作为“无用功能”删除：

- 数字人视频生成接口与测试入口
- OCR 自定义多模态接口
- TTS 自建服务和第三方服务配置
- SSO、LDAP 和外部用户映射
- 文档审批、密级和单文档 ACL
- 培训考试、证书和学习进度
- 部署检查、运行监控和备份恢复

处理原则：未配置时明确显示未启用，不影响主流程，也不对外暴露可消耗资源的接口。

## 9. 推荐执行顺序

### 第一阶段：生产安全加固

1. 关闭生产自由注册。
2. 删除默认管理员认证降级。

4. 增加登录限流和安全事件。
5. 保护 TTS 和系统资源接口。

### 第二阶段：数据一致性修复

1. 修复反馈和工单归属校验。

3. 修复培训考试事务与幂等性。

5. 修复看板权限和统计数据。

### 第三阶段：功能去重和结构整理

1. 合并知识管理重复入口。
2. 整合运营、反馈、工单和审计页面。
3. 删除登录页硬编码数据。

5. 明确 MySQL 为生产主数据库。

1. 建立独立测试数据库。
2. 补齐权限、并发、删除和上传测试。
3. 执行类型检查、生产构建和依赖审计。
4. 在测试环境完成完整业务回归。
5. 审查通过后启动 Winmail、企业微信和致远 OA 接入。

## 10. 开发目标定义

### 目标名称

**三期开发前安全加固、数据一致性修复与功能收敛**

### 目标描述

在不影响当前生产服务和远程数据库现有业务数据的前提下，修复本轮代码审查发现的 P0、P1 问题，完成关键重复功能收敛，建立独立自动化测试环境，为三期外部系统接入提供稳定、安全且可回归的基础版本。

### 完成条件

- 所有 P0 问题完成修复并通过测试。
- 所有 P1 问题完成修复，或有明确延期原因和隔离措施。
- 管理页面与 API 权限策略统一。
- 会话删除不再破坏审计链路。
- 考试提交具备事务和幂等保障。
- Excel 解析不再存在当前已知高危漏洞。
- 重复知识管理入口完成合并。
- 自动化测试不再连接生产数据库。
- `npm run typecheck` 通过。
- `npm run build` 通过。
- 生产依赖审计无高危和严重漏洞。
- 完成测试环境回归后再开始三期接口开发。

## 11. 实施约束

- 不直接修改或清空生产数据库现有数据。
- 数据库结构变更必须先备份并提供可回滚迁移脚本。
