# 西安天瑞汽车内饰件有限公司智能客服部署与运维手册

## 一、部署目标

将智能客服系统从本地开发环境迁移到公司内网服务器，供员工通过浏览器访问。当前推荐部署方式：

- 应用：Next.js 生产构建
- 数据库：公司自有 MySQL
- RAG：local_text，本地数据库分片轻量混合检索
- 对话模型：OpenAI-compatible 第三方 API
- 语音：OpenAI TTS 或自定义 TTS API

## 二、服务器要求

建议最低配置：

| 项目 | 建议 |
| --- | --- |
| CPU | 2 核及以上 |
| 内存 | 4GB 及以上 |
| 磁盘 | 50GB 及以上，视上传资料数量增加 |
| Node.js | 20 LTS 或更高 |
| MySQL | 8.0 或兼容版本 |
| 网络 | 可访问数据库和第三方模型 API |

## 三、环境变量

生产环境建议在 `.env.local` 配置：

```bash
DATABASE_PROVIDER=mysql
MYSQL_HOST=数据库地址
MYSQL_PORT=3306
MYSQL_DATABASE=数据库名
MYSQL_USER=用户名
MYSQL_PASSWORD=密码

RAG_PROVIDER=local_text
AI_CHAT_PROVIDER=custom
AI_CHAT_BASE_URL=https://模型服务地址/v1
AI_CHAT_API_KEY=模型APIKey
AI_CHAT_MODEL=模型ID

OCR_PROVIDER=none
OCR_API_URL=https://OCR服务地址
OCR_API_KEY=OCR_APIKey
OCR_AUTH_HEADER=Authorization
OCR_HEADERS=
OCR_REQUEST_FORMAT=multipart
OCR_FILE_FIELD=file
OCR_MODEL_FIELD=model
OCR_PROVIDER_FIELD=provider
OCR_PAYLOAD_TEMPLATE=
OCR_MODEL=OCR模型

TTS_PROVIDER=custom
TTS_API_URL=https://语音服务地址
TTS_STATUS_URL=https://语音状态地址/{task_id}
TTS_API_KEY=语音APIKey
TTS_AUTH_HEADER=Authorization
TTS_HEADERS=
TTS_PAYLOAD_TEMPLATE=
TTS_MODEL=语音模型
TTS_VOICE=音色

DIGITAL_HUMAN_PROVIDER=none
DIGITAL_HUMAN_API_URL=https://数字人生成服务地址
DIGITAL_HUMAN_STATUS_URL=https://数字人状态服务地址/{job_id}
DIGITAL_HUMAN_API_KEY=数字人APIKey
DIGITAL_HUMAN_AUTH_HEADER=Authorization
DIGITAL_HUMAN_HEADERS=
DIGITAL_HUMAN_PAYLOAD_TEMPLATE=
DIGITAL_HUMAN_MODEL=数字人模型
DIGITAL_HUMAN_AVATAR_ID=形象ID
DIGITAL_HUMAN_VOICE_ID=音色ID

SSO_PROVIDER=none
SSO_AUTHORIZE_URL=https://统一身份平台/oauth2/authorize
SSO_TOKEN_URL=https://统一身份平台/oauth2/token
SSO_USERINFO_URL=https://统一身份平台/oauth2/userinfo
SSO_CLIENT_ID=客户端ID
SSO_CLIENT_SECRET=客户端密钥
SSO_SCOPES=openid profile email
SSO_DEFAULT_DEPARTMENT=综合管理部

LDAP_PROVIDER=none
LDAP_URL=ldaps://AD服务器地址:636
LDAP_BIND_DN=只读服务账号DN
LDAP_BIND_PASSWORD=只读服务账号密码
LDAP_SEARCH_BASE=用户搜索BaseDN
LDAP_SEARCH_FILTER=(|(mail={{login}})(uid={{login}})(sAMAccountName={{login}}))
LDAP_USER_DN_TEMPLATE=
LDAP_EMAIL_ATTRIBUTE=mail
LDAP_NAME_ATTRIBUTE=displayName
LDAP_DEPARTMENT_ATTRIBUTE=department
LDAP_POSITION_ATTRIBUTE=title
LDAP_DEFAULT_DOMAIN=公司邮箱域

AUTH_SECRET=请填写随机长字符串
APP_BASE_URL=http://服务器IP:3000
NOTIFICATION_WEBHOOK_URL=
NOTIFICATION_EMAIL_WEBHOOK_URL=
NOTIFICATION_WECOM_WEBHOOK_URL=
MAX_UPLOAD_MB=20
```

如果暂时不启用服务端语音，可先保留 TTS 配置为空；员工端回答和培训讲稿会优先尝试服务端 TTS，失败后退回浏览器本地朗读，不影响知识问答和课程学习。
配置语音后，可在 `/admin/settings` 的“TTS 语音试听”中输入测试文本并播放返回音频。自定义 TTS 支持直接音频流、base64、音频 URL，以及配置 `TTS_STATUS_URL` 后的异步任务轮询。
如果暂时不启用数字人，将 `DIGITAL_HUMAN_PROVIDER` 保持为 `none`，PPT 培训仍可使用讲稿和语音讲解。
配置数字人后，可在 `/admin/settings` 的“数字人接口测试”提交短测试讲稿，确认第三方接口是否返回任务编号或视频地址。该测试不写入课程任务表，但第三方平台可能会创建短测试任务。
如需识别扫描件 PDF 或图片资料，在配置页填写 `OCR_PROVIDER=custom`、`OCR_API_URL`、`OCR_API_KEY` 后，可通过 `/admin/settings` 的“OCR 实际测试”上传样例文件验证。验证通过后，管理员可在 `/admin/documents` 直接上传扫描件 PDF 或图片资料，系统会调用同一套 OCR 逻辑生成知识分片。
第三方 OCR/TTS/数字人接口默认使用 `Authorization: Bearer APIKey`。如果服务商要求 `X-API-Key`、`api-key` 或其他认证头，可分别配置 `OCR_AUTH_HEADER`、`TTS_AUTH_HEADER`、`DIGITAL_HUMAN_AUTH_HEADER`；额外请求头可用对应 `*_HEADERS` JSON 配置，值中支持 `{{api_key}}` 占位。若服务商请求体不同，可用 `TTS_PAYLOAD_TEMPLATE`、`OCR_PAYLOAD_TEMPLATE`、`DIGITAL_HUMAN_PAYLOAD_TEMPLATE` 配置 JSON 模板。
管理员也可以先在 `/admin/settings` 的“服务商预设模板”中一键套用常见模式，再补真实 API URL、API Key、模型、音色或数字人形象。
如需企业统一登录，将 `SSO_PROVIDER` 设置为 `oidc`，并在统一身份平台配置回调地址：`{APP_BASE_URL}/api/auth/sso/callback`。
如需直连企业 LDAP 或 AD，将 `LDAP_PROVIDER` 设置为 `custom`，并选择“服务账号搜索 + 用户绑定”或 `LDAP_USER_DN_TEMPLATE` 直接绑定。LDAP 登录成功后会自动同步员工邮箱、姓名、部门和岗位；系统内被禁用的账号不会被自动启用。
培训课程生成后默认未发布。管理员确认讲稿、语音或数字人视频效果后，在 `/admin/training` 发布课程，员工才会在 `/training` 看到课程；下架或归档后员工端不再展示。

## 四、安装与启动

首次部署：

```bash
npm install
npm run typecheck
npm run build
npm run start
```

推荐使用项目内置脚本完成首次部署和后续升级：

```bash
chmod +x scripts/deploy-local.sh scripts/backup-mysql.sh
npm run deploy:local
```

脚本会检查 `.env.local`、安装依赖、执行类型检查、构建生产产物，并通过 `ecosystem.config.cjs` 使用 PM2 启动或重载 `tianrui-ai-support`。

默认生产端口为 `3000`。访问：

```text
http://服务器IP:3000
```

全新数据库首次启动前，在 `.env.local` 临时设置初始管理员：

```env
INITIAL_ADMIN_EMAIL=admin@example.com
INITIAL_ADMIN_PASSWORD=请使用独立强密码
INITIAL_ADMIN_NAME=系统管理员
```

首次管理员创建成功后，从 `.env.local` 删除 `INITIAL_ADMIN_PASSWORD`。随后在“用户管理”中创建正式管理员和员工账号，并维护员工部门和岗位。知识库可按全员、部门、岗位或仅管理员控制可见范围，单份资料还可配置密级、发布状态、部门/岗位/用户 ACL。

## 六、Nginx 反向代理

如需用 `http://客服系统地址` 访问，可配置 Nginx：

```nginx
server {
  listen 80;
  server_name your-domain-or-ip;

  client_max_body_size 50m;

  location / {
    proxy_pass http://127.0.0.1:3000;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }
}
```

## 七、部署前检查

上线前在系统中打开：

```text
/admin/deploy
```

重点确认：

- 数据库连接为 ready
- MySQL 核心表已创建
- AUTH_SECRET 已配置
- 对话模型 API 已配置
- RAG_PROVIDER 为 local_text
- 至少有 1 个管理员账号
- 至少有 1 个可用知识库
- 至少有 3 份已发布且可检索资料
- 至少有 1 个已发布培训课程（如本期启用培训场景）
- `/admin/insights` 中无未处理高危安全告警
- `/admin/insights` 的“安全审计”可查看敏感输入、提示词注入和异常访问事件
- 连续触发敏感输入、提示词注入或无权限知识库访问时，会在 15 分钟窗口内生成异常访问告警
- 试运行验收得分建议不低于 80%

资料上传后默认为草稿。上线前请在“知识管理”的“权限治理”中完成“提交审核 -> 审核发布”，未发布资料不会进入员工端检索。
如误改资料标题、文件类型、处理状态或本地 RAG 正文，可在“资料版本记录”点击“回滚”恢复历史版本。新版本会保存 `document_version_chunks` 正文快照；旧版本若没有快照，回滚时会保留当前 `document_chunks` 并在版本备注中提示。

## 八、备份方案

### 1. MySQL 备份

建议每日定时备份：

```bash
mysqldump -h 数据库地址 -P 3306 -u 用户名 -p 数据库名 > backup-$(date +%F).sql
```

也可以使用项目内置脚本，脚本会生成压缩备份并清理过期文件：

```bash
MYSQL_HOST=数据库地址 \
MYSQL_PORT=3306 \
MYSQL_DATABASE=数据库名 \
MYSQL_USER=用户名 \
MYSQL_PASSWORD=密码 \
BACKUP_DIR=/data/tianrui-ai-support/backups \
KEEP_DAYS=14 \
npm run backup:mysql
```

可加入服务器 `crontab` 每天凌晨执行：

```cron
10 2 * * * cd /data/tianrui-ai-support && /usr/bin/env bash scripts/backup-mysql.sh >> /data/tianrui-ai-support/backups/backup.log 2>&1
```

建议保留：

- 最近 7 天每日备份
- 最近 4 周每周备份
- 每月 1 份长期归档

### 2. 恢复数据库

生产环境先使用项目脚本做非破坏性恢复验证。脚本优先恢复到独立临时数据库；账号无建库权限时使用 `_rv_` 临时表前缀，按备份文件记录的每表快照行数核对，成功后自动清理临时对象，不覆盖生产表：

```bash
npm run verify:restore:mysql
```

只有灾难恢复时才手工覆盖目标库：

```bash
mysql -h 数据库地址 -P 3306 -u 用户名 -p 数据库名 < backup-2026-07-04.sql
```

### 3. 配置文件备份

务必备份 `.env.local`，其中包含数据库、模型 API 和登录密钥。不要通过聊天、邮件或公开文档传播该文件。

## 九、升级流程

### 1. 发布前检查

```bash
npm run deploy:preflight
```

预检会阻断以下情况：生产密钥或地址无效、MySQL 不可连接或缺核心表、磁盘不足、备份/恢复记录过期、登录页或鉴权接口异常。生产环境应设置 `MYSQL_AUTO_MIGRATE=false`，数据库结构变更需要先单独审核执行。

### 2. 分阶段发布

准备不包含 `node_modules`、`.next` 和生产数据目录的 `release.tar.gz`，然后执行：

```bash
npm run deploy:package -- /absolute/path/to/release.tar.gz
```

脚本在 `tianrui-releases/staging-*` 中安装依赖、类型检查、构建和预检；全部成功后才生成 `rollback-*.tar.gz`、同步生产代码并只重载 `tianrui-ai-support`。健康检查要求 `/login=200`、`/api/auth/me=401`；失败会自动恢复切换前版本。

以下路径不会被发布或回滚覆盖、删除：

- `.env.local`
- `.data`
- `uploads`
- `backups`
- `.ops`

默认仅保留最近 2 个普通回滚包，可通过 `RELEASE_KEEP_ROLLBACKS` 调整，避免占满磁盘。

### 3. 快速回滚

回滚最近版本：

```bash
npm run deploy:rollback
```

回滚指定版本：

```bash
npm run deploy:rollback -- /home/ubuntu/tianrui-releases/rollback-YYYYMMDD-HHMMSS.tar.gz
```

回滚前仍会保存当前版本；回滚后的健康检查失败时，会自动恢复回滚前版本。任何脚本都只允许管理 PM2 应用 `tianrui-ai-support`，且会校验 PM2 `cwd` 与 `APP_DIR` 一致。

升级后检查：

- `/admin/deploy`
- `/admin/pilot`
- 员工端 `/chat`
- 管理员端 `/admin/documents`

## 十、自动化部署流水线

项目已经包含两个 GitHub Actions：

- `.github/workflows/ci.yml`：推送或 PR 到 `main/master` 时自动执行 `npm ci`、类型检查、生产构建和 Playwright 回归。
- `.github/workflows/deploy.yml`：手动触发或推送 `v*` tag 时，先执行完整验证，再通过 SSH 登录服务器发布。接入仓库后应将远程步骤升级为制作发布包并调用 `deploy:package`。

启用自动部署前，需要在 GitHub 仓库 Settings -> Secrets and variables -> Actions 中配置：

| Secret | 说明 |
| --- | --- |
| `DEPLOY_HOST` | 服务器 IP 或域名 |
| `DEPLOY_PORT` | SSH 端口，默认可填 `22` |
| `DEPLOY_USER` | SSH 用户 |
| `DEPLOY_SSH_KEY` | 可登录服务器的私钥 |
| `DEPLOY_APP_DIR` | 服务器上的项目目录，例如 `/data/tianrui-ai-support` |

服务器侧准备：

1. 安装 Node.js 20+、Git、PM2、MySQL 客户端。
2. 将项目 clone 到 `DEPLOY_APP_DIR`。
3. 在 `DEPLOY_APP_DIR/.env.local` 填写生产环境变量。
4. 确认服务器可以访问 MySQL、模型 API、OCR/TTS/数字人 API。
5. 首次启动运行 `npm run deploy:local`；后续运行 `npm run deploy:preflight` 和一次受控 `deploy:package`，确认回滚包与健康检查正常。

日常发布建议：

1. 代码合并到 `main` 后等待 CI 通过。
2. 在 GitHub Actions 手动运行 `Deploy`，填写要部署的分支或 tag。
3. 或创建版本 tag，例如 `v1.0.0`，推送后自动触发部署。
4. 部署完成后打开 `/admin/deploy` 和员工端 `/chat` 做冒烟验证。

## 十一、常见问题

### 1. 页面能打开，但问答不可用

检查：

- `AI_CHAT_PROVIDER`
- `AI_CHAT_BASE_URL`
- `AI_CHAT_API_KEY`
- `AI_CHAT_MODEL`
- 第三方模型 API 是否允许服务器访问

### 2. 上传资料后没有引用

检查：

- 资料状态是否为 ready
- `RAG_PROVIDER` 是否为 local_text
- 文档是否解析出文字
- `/admin/qa-tests` 是否能跑出引用

### 3. 数字人视频生成失败

检查：

- `DIGITAL_HUMAN_PROVIDER` 是否为 custom
- `DIGITAL_HUMAN_API_URL` 和 `DIGITAL_HUMAN_API_KEY` 是否已配置
- 认证方式是否匹配服务商要求，必要时配置 `DIGITAL_HUMAN_AUTH_HEADER` 或 `DIGITAL_HUMAN_HEADERS`
- 请求体是否匹配服务商要求，必要时配置 `DIGITAL_HUMAN_PAYLOAD_TEMPLATE`
- 第三方接口是否允许服务器访问
- 返回结果是否包含 `video_url`，或包含 `job_id` 且已配置 `DIGITAL_HUMAN_STATUS_URL`
- `/admin/settings` 的“数字人接口测试”是否能返回任务编号、状态或视频地址
- `/admin/training` 中课程是否已生成讲稿

### 4. OCR 识别失败

检查：

- `OCR_PROVIDER` 是否为 custom
- `OCR_API_URL`、`OCR_API_KEY` 是否填写
- 认证方式是否匹配服务商要求，必要时配置 `OCR_AUTH_HEADER` 或 `OCR_HEADERS`
- 请求格式是否正确，multipart 字段名可配置 `OCR_FILE_FIELD` / `OCR_MODEL_FIELD`，JSON base64 模式需设置 `OCR_REQUEST_FORMAT=json_base64`
- 请求体是否匹配服务商要求，必要时配置 `OCR_PAYLOAD_TEMPLATE`
- 第三方 OCR 是否支持 `multipart/form-data` 的 `file` 字段
- 返回 JSON 是否包含 `text`、`pages` 或 `results`
- 上传文件是否为图片或扫描件 PDF，且未超过 `MAX_UPLOAD_MB`

### 5. TTS 语音生成失败

检查：

- `TTS_PROVIDER` 是否为 openai 或 custom
- 使用 custom 时，`TTS_API_URL`、`TTS_API_KEY` 是否填写
- 认证方式是否匹配服务商要求，必要时配置 `TTS_AUTH_HEADER` 或 `TTS_HEADERS`
- 请求体是否匹配服务商要求，必要时配置 `TTS_PAYLOAD_TEMPLATE`
- 异步任务是否返回 `task_id` / `job_id`，并配置了 `TTS_STATUS_URL`
- 返回结果是否为音频流、音频 URL 或 base64 音频字段
- `/admin/settings` 的“TTS 语音试听”是否能播放测试音频

### 6. 企业统一登录失败

检查：

- `SSO_PROVIDER` 是否为 oidc
- 授权地址、Token 地址、用户信息地址是否填写完整
- `SSO_CLIENT_ID`、`SSO_CLIENT_SECRET` 是否正确
- 身份平台回调地址是否等于 `{APP_BASE_URL}/api/auth/sso/callback`
- 用户信息接口是否返回 `sub` 和有效 `email`

### 7. LDAP / AD 登录失败

检查：

- `LDAP_PROVIDER` 是否为 custom
- `LDAP_URL` 是否能从应用服务器访问，内网防火墙是否放通 389 或 636
- 使用服务账号搜索时，`LDAP_BIND_DN`、`LDAP_BIND_PASSWORD`、`LDAP_SEARCH_BASE` 是否正确
- 使用用户 DN 模板时，`LDAP_USER_DN_TEMPLATE` 是否能渲染出真实用户 DN
- `LDAP_SEARCH_FILTER` 是否匹配企业目录中的邮箱、uid 或 sAMAccountName
- 登录用户是否能返回有效邮箱，或是否已配置 `LDAP_DEFAULT_DOMAIN`
- 系统内同邮箱账号是否已被管理员禁用

### 8. 登录失败

检查：

- MySQL 是否可连接
- `AUTH_SECRET` 是否被更换
- 用户状态是否为 active

### 9. 构建成功但访问异常

执行：

```bash
pm2 logs tianrui-ai-support
npm run typecheck
npm run build
```

优先查看服务日志中的具体报错。
